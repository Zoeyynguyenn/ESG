자원 관리 :프로젝트
•프로젝트: 미리 할당된 자원을 공유하는 최소한의 단위–’사용자 그룹’, ‘팀’ 또는 ‘프로젝트 그룹’ 등으로 사용됩니다.–사용자는 하나의 프로젝트 하위, 하나의 자원 그룹에 속할 수 있습니다✓NIPA-A100L 또는 NIPA-A100Q 또는 NIPA-H100E 또는 NIPA-H100Q...
NOTE실제 프로젝트 이름 및 접근가능한 프로젝트 수, 그에 따른 자원량은 정책에 따라 변경될 수 있습니다.

--- page 10 ---
자원 관리 :자원그룹
•자원 그룹: 하드웨어 리소스를 관리하는 단위 그룹–프로젝트별로 자원 그룹이 할당됩니다.–사용자는 지정된 자원 그룹 내의 자원만을 이용해서 연산 세션을 생성할 수 있습니다.–사용자는할당된자원그룹내자원을자유롭게사용할수있습니다.예시) NIPA-H100E프로젝트에H100E자원그룹을할당하고, NIPA-A100Q프로젝트에A100Q자원그룹을할당합니다.
A 프로젝트에소속된사용자는H100 GPU를사용하게되고,B 프로젝트에소속된사용자는A100 GPU를사용하게됩니다.

저장공간 다루기 :파일 업로드
드래그앤 드랍으로 4GB 이상 파일/폴더 업로드는 불가합니다. 따라서 해당 크기를 초과하는 파일/폴더 업로드는 파일 브라우저, SFTP 세션을 활용하시기 바랍니다.
파일 업로드는 세가지 방법 중 하나를 선택합니다.폴더 탐색기에서 드래그 앤 드랍, 파일 브라우저 사용, 그리고 SFTP 세션으로 대용량 파일 업로드 방법이 있습니다.

--- page 16 ---
저장공간 다루기 :파일 업로드
파일 브라우저로 파일 업로드/다운로드 할 경우 Execute filebrowser버튼을 클릭합니다. 이후 약간의 시간이 지나면 파일 브라우저 이미지로 생성된 세션에서 띄워진 파일 브라우저 웹 앱이 새 창으로 열리게 됩니다.

--- page 17 ---
저장공간 다루기 :파일 업로드
해당 파일브라우저 웹앱을 실행하기 위해서는 최소한의 자원(세션 개수 포함)이 필요합니다. 세션에 할당할 수 있는 최소한의 여유 자원이 없는 경우(세션 개수 포함)는 파일브라우저를 활용한 업로드/다운로드가 불가합니다.

--- page 18 ---
저장공간 다루기 :파일 업로드
SFTP 세션을 활용하여 대용량 파일 업로드/다운로드 할 경우, Run SFTP Server 버튼을 클릭합니다. 이후 약간의 시간이 지나면 SFTP 서버에 접속할 수 있는 SSH 키 정보와 명령어를 확인할 수 있는 창이 띄워집니다. 해당 창의 내용을 참고하여 파일 업로드/다운로드가 가능합니다. 단, SFTP 세션 역시 세션이기 때문에, 세션에 할당할 수 있는 최소한의 여유 자원이 없는 경우(세션 개수 포함)는 SFTP 대용량 서버를 활용한 업로드/다운로드가 불가합니다.

세션에 대해 자주하는 질문
Q. 제공되는 환경이 일반 VM 이나 Baremetal환경이 아닌 컨테이너 기반의 서비스라고 설명 들었는데 차이점이 뭔가요?A. 저희가 제공하는 GPU 학습 환경은 도커 컨테이너(Docker container) 기반의 서비스 입니다.도커 컨테이너는 애플리케이션과 그 종속성을 하나의 패키지로 묶어주는 기술입니다. 컨테이너는 호스트 운영 체제 위에서 실행되며, 별도의 운영 체제를 필요로 하지 않습니다. 학습 환경에서 도커 컨테이너를 사용할 경우 VM 환경보다 쉽고 빠르게 자원 확장, 축소가 가능하므로 효율적인 관리가 가능합니다. 특히 딥러닝 및 고성능 컴퓨팅 연산 자원관리에 최적화 되어 있습니다. 가령 장애 발생 시 중단된 시점부터 재시작하거나 동일 인스턴스를 시작하는 등, 다양한 장애 대응 옵션을 제공합니다. 따라서 성능 확보가 중요한 딥러닝 학습, 서비스 환경에서는 도커 컨테이너를 활용해 빠르게 할당/회수하는 시스템을 추천합니다.

세션에 대해 자주하는 질문
Q. 제공받은 서비스에 도커(Docker)가 설치된 것으로 확인되는데 도커 데몬(docker daemon) 이 돌아가지 않습니다. 제공 받은 환경에서 여러 개의 컨테이너를 띄워서 학습하려고 하는데 도움 부탁드립니다.A. 저희가 제공하는 환경은 도커 컨테이너(Docker container) 기반의 환경입니다. 사용하고 계시는 환경은 이미 컨테이너 내부에서 작업하는 것으로 도커-인-도커(docker-in-docker) 는 로우 레벨(low level)의 기술적 문제를 많이 일으키고 성능/보안 상 권장되지 않으며 해당 작업으로 인한 기술적인 오류/결함은 지원하지 않습니다.
